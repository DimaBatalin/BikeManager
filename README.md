# BikeManager

Telegram-бот для учёта ремонтов велосипедов в мастерской: создание заявок,
редактирование, архивация, отчёты по выручке и быстрое добавление
разовых («фиктивных») ремонтов на произвольную сумму.

## Возможности

- **Создание ремонта** — пошаговая FSM-форма: ФИО клиента → источник заявки →
  контакт → тип велосипеда (механический/электро) → название/поломки →
  стоимость → примечания.
- **Быстрый (фиктивный) ремонт** — команда `/fake_repair`, позволяющая
  добавить ремонт на произвольную сумму в один-два шага, без прохождения
  полной формы (используется для разовых услуг типа «просто подкачать
  колесо»). Подробнее — см. раздел [«Быстрый ремонт»](#быстрый-ремонт).
- **Редактирование** — изменение любого поля активного ремонта (ФИО, контакт,
  источник, тип велосипеда, поломки, стоимость, примечания, дата).
- **Архивация** — закрытие ремонта переносит его из активных в архив с
  проставлением даты закрытия; архив можно листать постранично, фильтровать
  по источнику, восстанавливать записи обратно в активные, менять дату
  архивации или удалять запись навсегда.
- **Отчёты** — сводки по количеству ремонтов и выручке за последние
  4 недели или 12 месяцев, с фильтрацией по источнику заявок.
- **Разграничение доступа** — бот отвечает только пользователям из белого
  списка `ALLOWED_USER_IDS`.
- **Отмена диалога** — команда `/cancel` сбрасывает любой незавершённый
  диалог (создание/редактирование ремонта и т.д.).

## Стек технологий

- Python 3.11+
- [aiogram 3.19](https://docs.aiogram.dev/) — асинхронный фреймворк для
  Telegram Bot API, long polling
- Хранилище данных — JSON-файлы (без внешней БД), потокобезопасный доступ
  через `threading.RLock`
- Хранение состояния FSM — в оперативной памяти (`aiogram.fsm.storage.memory.MemoryStorage`)
- Логирование — стандартный модуль `logging` с ротацией файлов

## Установка и запуск

1. Клонируйте репозиторий и создайте виртуальное окружение:

   ```bash
   git clone https://github.com/DimaBatalin/BikeManager.git
   cd BikeManager
   python3 -m venv venv
   source venv/bin/activate  # Windows: venv\Scripts\activate
   ```

2. Установите зависимости:

   ```bash
   pip install -r requirements.txt
   ```

3. Создайте файл `config.py` на основе шаблона `config.example.py`:

   ```bash
   cp config.example.py config.py
   ```

   и заполните его реальными значениями (см. раздел ниже).

4. Запустите бота:

   ```bash
   python bot.py
   ```

## Настройка `config.py`

Файл `config.py` не хранится в репозитории (добавлен в `.gitignore`), так как
содержит токен бота и список ID пользователей. Используйте
`config.example.py` как основу. Требуются следующие переменные:

| Переменная                     | Тип               | Назначение                                                                 |
|---------------------------------|-------------------|-----------------------------------------------------------------------------|
| `TG_TOKEN`                      | `str`             | Токен бота, выданный [@BotFather](https://t.me/BotFather)                   |
| `ALLOWED_USER_IDS`               | `set[int]`        | Telegram ID пользователей, которым разрешён доступ к боту                   |
| `ACTIVE_PATH`                    | `pathlib.Path`    | Путь к JSON-файлу активных ремонтов (по умолчанию `data/active_repairs.json`) |
| `ARCHIVE_PATH`                   | `pathlib.Path`    | Путь к JSON-файлу архива (по умолчанию `data/archive_repairs.json`)          |
| `REPAIR_SOURCES`                 | `dict[str, str]`  | Источники заявок: ключ (используется в данных) → отображаемое название      |
| `ELECTRIC_BIKE_BREAKDOWNS_PATH`  | `list[str]`       | Список типовых поломок для электровелосипедов (для клавиатуры с чекбоксами) |

### Токен бота и права доступа

- Токен получается у [@BotFather](https://t.me/BotFather) командой `/newbot`.
- Доступ к боту ограничен списком `ALLOWED_USER_IDS` — все остальные
  пользователи получат сообщение об отказе в доступе
  (`middlewares/access_control.py`). Свой Telegram ID можно узнать, например,
  у [@userinfobot](https://t.me/userinfobot).

## Быстрый ремонт

Команда `/fake_repair` создаёт ремонт без прохождения полной формы:

- `/fake_repair 1500` — ремонт с суммой 1500 руб. создаётся сразу.
- `/fake_repair` (без аргумента) — бот отдельным сообщением спросит сумму.

Остальные поля заполняются значениями по умолчанию: `FIO="Без имени"`,
`contact="-"`, `namebike="-"`, `breakdowns=[]`, `notes="Быстрый ремонт"`,
`repair_type="quick"`. Такая запись полностью совместима с остальной системой:
её можно редактировать, закрывать (архивировать), она попадает в отчёты и
фильтруется по источнику `quick` — задайте отображаемое имя для этого ключа
в `REPAIR_SOURCES` (например, `"quick": "Быстрый ремонт"`).

Сумма валидируется: принимается только целое положительное число, при
некорректном вводе бот попросит ввести значение повторно.

## Формат хранения данных

`data/active_repairs.json` и `data/archive_repairs.json` — списки JSON-объектов
одинаковой структуры. Пример записи активного ремонта:

```json
{
  "id": 42,
  "FIO": "Иванов Иван Иванович",
  "contact": "+79991234567",
  "repair_type": "avito",
  "isMechanics": true,
  "namebike": "Stels Navigator",
  "breakdowns": ["Порвана цепь 500", "Прокол колеса 200"],
  "cost": 700,
  "notes": "-",
  "date": "05.07.2026"
}
```

При закрытии ремонта (архивации) в объект добавляется поле `archive_date`
(дата в формате `ДД.ММ.ГГГГ`), а сам объект переносится из
`active_repairs.json` в `archive_repairs.json`. При восстановлении из архива
поле `archive_date` удаляется, а объект переносится обратно.

Запись файлов производится атомарно (через временный файл и `rename`), чтобы
сбой посреди записи не привёл к повреждению данных.

## Логи

Логи пишутся в `logs/bot.log` с ежедневной ротацией (`TimedRotatingFileHandler`,
ротация в полночь, хранится 7 бэкапов) и одновременно выводятся в консоль.
Формат записи:

```
2026-07-05 12:00:00,000 - INFO - handlers.create_new_repair - Создан новый ремонт ID:42. ...
```

Каждый модуль (`handlers/*`, `services/*`, `middlewares/*`) использует
собственный именованный логгер (`logging.getLogger(__name__)`), что позволяет
видеть, из какой части приложения пришла запись. Основные уровни:

- **DEBUG** — детали, не относящиеся к бизнес-логике напрямую.
- **INFO** — бизнес-события: создание/редактирование/архивация/восстановление
  ремонта, генерация отчётов.
- **WARNING** — подозрительные ситуации: некорректный ввод пользователя,
  попытки доступа неавторизованных пользователей, повреждённые записи данных.
- **ERROR/CRITICAL** — сбои чтения/записи файлов, необработанные исключения
  (логируются с полным traceback через `logger.exception`).

Контактные данные клиентов в логах частично маскируются
(`utils.formatter.mask_contact`), чтобы не хранить их в открытом виде.

## Известные ограничения

- Состояние FSM хранится в оперативной памяти (`MemoryStorage`) — при
  перезапуске бота все незавершённые диалоги (создание/редактирование
  ремонта) будут потеряны, пользователю нужно будет начать заново или
  использовать `/cancel`.
- Хранилище данных — плоские JSON-файлы, а не полноценная БД: подходит для
  небольшой мастерской, но не рассчитано на большие объёмы данных или высокую
  конкурентную нагрузку (см. `TODO.txt`, пункт «Разобраться со случаем
  больших данных»).
- Список актуальных доработок и известных задач — в файле `TODO.txt`.
